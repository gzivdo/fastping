# fastping

📖 **[Documentation in English →](README.md)**

Stateless-утилита массового ICMP-пинга. Из одного raw-сокета выстреливает весь
батч echo-запросов, а вторым сокетом ловит ответы — без блокировок между ними —
после чего повторно шлёт **только молчунам**, несколько раз. Та же асинхронная
архитектура «отправитель/приёмник», что у `zmap`/`masscan`, но сверху —
адаптивная очередь ретраев по каждому хосту, которую массовые сканеры сознательно
не делают.

* Один статический бинарник, ~380 КБ.
* На горячем пути нет таблицы состояний: индекс цели, проверочный cookie и
  таймстемп отправки зашиты в payload ICMP и возвращаются хостом обратно — одного
  ответного пакета достаточно, чтобы понять, чей он и какой RTT.
* Режимы вывода: `plain`, `fping`-совместимый, `zabbix` (нативный протокол
  sender) и `json` (для Zabbix LLD / Nagios). Готовая интеграция с Zabbix и Nagios.
* **Dual-stack** IPv4 + IPv6; цели — литералы, хостнеймы (A/AAAA), IPv4-CIDR
  или файлы.

## Идея на пальцах

Обычный `ping` идёт по хостам по одному: отправил — ждёшь ответа (или таймаута) —
следующий. `fping` умнее, он перемежает много хостов по кругу, но всё равно гонит
всё через один цикл процесса. На тысячах хостов почти всё время уходит на
**ожидание**.

fastping вместо этого **отправляет все письма сразу**: один поток выстреливает
весь батч echo-запросов, второй просто ловит, что вернулось. Ведётся чек-лист,
кто ответил; после короткой паузы повторно шлём **только молчунам**, пару раз.
Итоговое время ≈ «один таймаут плюс пара ретраев для отставших», а не «таймаут ×
количество хостов».

Хитрость, которая делает приёмник дешёвым: каждый зонд несёт индекс своей цели,
проверочный cookie и таймстемп отправки **внутри payload ICMP**, и хост эхом
возвращает их. Поэтому один ответный пакет самодостаточен — приёмнику не нужна
общая таблица, чтобы понять, чей это ответ и сколько он шёл.

## Сравнение с альтернативами

| Инструмент | Модель | Адаптивный ретрай | loss/RTT по хосту | Вывод для мониторинга | Силён в |
|---|---|---|---|---|---|
| `ping` | 1 хост, блокирующий | — | да | — | один хост |
| `fping` | много хостов, round-robin, 1 сокет | слепой повтор | да | текст | свипы по LAN, скрипты |
| `nmap -sn` | discovery, богатый | да | ограниченно | XML/grep | discovery + доп. зонды |
| `zmap` | stateless, интернет-масштаб | **нет** (fire-and-forget) | нет (survey) | нужна постобработка | разовые интернет-опросы |
| `masscan` | stateless, упор на TCP | нет | нет | списки | сканы портов по интернету |
| **fastping** | **stateless-батч + адаптивный ретрай** | **да** | **да** | **Zabbix/Nagios/JSON из коробки** | **повторяющиеся свипы множества известных хостов** |

**Чем отличается и почему лучше в своей нише:**

* **против `fping`** — та же семантика «жив + потери + RTT с ретраями», но
  zmap-style асинхронный батч масштабируется на куда большее число хостов, а
  бэкенд AF_PACKET + `PACKET_RX_RING` перестаёт терять ответы под нагрузкой (а
  потерянный ответ в `fping` выглядит как ложный «down»). Плюс нативный вывод в
  Zabbix/Nagios/JSON — парсить нечего.
* **против `zmap`/`masscan`** — они быстрее для разовых интернет-опросов, но
  намеренно *fire-and-forget*: нет ретрая по хосту, нет удобных loss/RTT, и ты
  разгребаешь дамп результатов. fastping сохраняет их stateless горячий путь, но
  добавляет ровно то, что нужно мониторингу: адаптивную очередь ретраев и готовые
  метрики по каждому хосту.
* **Ниша:** есть N *известных* хостов (инвентарь Zabbix/Nagios, пара /16) и нужно
  обходить их **часто, быстро и надёжно**, отдавая alive/loss/RTT прямо в
  мониторинг — без раздувания «процесс-на-хост» и без тяжёлого сканера.

**Честные ограничения:** только ICMP echo; цели — IPv4 **или IPv6** (дефолтный
сокетный бэкенд dual-stack; бэкенд AF_PACKET — только IPv4). Раскрытие IPv6-CIDR
намеренно не поддержано — перечисляй адреса явно. Cookie на FNV (быстрый, не
криптографический — норм для LAN/мониторинга, не для защиты от спуфинга во
враждебном интернете пока); бэкенд AF_PACKET ходит через MAC шлюза (свипы за
пределами подсети), его TX ещё не zero-copy; не оптимизирован под истинный
интернет-масштаб как zmap (нет PF_RING / multi-queue). См. roadmap.

## Бэкенды

Два бинарника, одинаковый CLI и режимы вывода (общее ядро в `src/lib.rs`):

| Бинарник | Транспорт | Когда |
|---|---|---|
| `fastping` | raw ICMP + ICMPv6 сокеты (ядро строит IP, делает routing+ARP/ND) | по умолчанию; **dual-stack** (IPv4+IPv6), любой канал, включая loopback и свою подсеть |
| `fastping-afpacket` | AF_PACKET TX + mmap `PACKET_RX_RING` | высокий pps, свипы за пределами подсети; не теряет ответы в буфере сокета; **только IPv4** |

`fastping-afpacket` адресует кадры на **MAC шлюза по умолчанию** (то есть для
маршрутизируемых свипов; своя подсеть/loopback → `fastping`). Нужна заполненная
ARP-запись шлюза (`ping <gw>` один раз, если её нет) и `-i/--iface` для выбора
интерфейса.

## Сборка

```sh
cargo build --release
# raw-сокетам нужен CAP_NET_RAW — либо от root, либо выдать capability разово:
sudo setcap cap_net_raw+ep ./target/release/fastping
sudo setcap cap_net_raw+ep ./target/release/fastping-afpacket
```

## Использование

```
fastping [ОПЦИИ] [IP...]

ЦЕЛИ (можно комбинировать; без аргументов читается stdin):
    IP...                 IPv4/IPv6-литералы или хостнеймы (резолв в A/AAAA)
    -f, --file <path>     файл со строками "ip|host" или "ip|host имя"
    -c, --cidr <cidr>     раскрыть IPv4-CIDR (можно несколько раз; IPv6 нельзя)

ОПЦИИ:
    -t, --timeout <ms>    ожидание раунда до ретрая         [по умолч. 1500]
    -r, --retries <n>     доп. раунды для молчунов           [по умолч. 2]
        --rate <pps>      темп отправки на весь батч, 0=макс [по умолч. 0]
        --payload <n>     байт payload ICMP (мин. 20)        [по умолч. 20]
    -o, --output <mode>   plain | fping | zabbix | json     [по умолч. plain]
    -C, --count <n>       проб на хост (режимы с фикс. пробами)[по умолч. 3]
        --discover        zabbix/json: адаптивный ретрай (жив/мёртв) вместо -C
        --key <prefix>    префикс ключа item в режиме zabbix [по умолч. icmp]
        --server <addr>   сервер/прокси Zabbix host[:10051]
        --batch <n>       значений на одну sender-сессию, 0=все [по умолч. 1000]
    -i, --iface <name>    исходящий интерфейс (только бэкенд afpacket)
```

### Примеры

```sh
# discovery: жив/мёртв + первый RTT, ретрай только молчунам
fastping -c 10.0.0.0/16 --rate 50000 -t 1000 -r 2

# fping-совместимый: N проб каждому хосту, '-' — потеря
cat hosts.txt | fastping -o fping -C 3

# zabbix: пуш прямо на сервер/прокси по нативному протоколу sender
fastping -f hosts.txt -o zabbix --key icmp -C 3 --server zbx-proxy:10051

# zabbix: вместо пуша вывести строки для zabbix_sender -i -
fastping -f hosts.txt -o zabbix --key icmp | zabbix_sender -z zbx -i -
```

Файл целей принимает `ip` или `ip имя`. Если задано имя — режим zabbix берёт его
как имя хоста в Zabbix; иначе используется строка IP.

## Режимы подробно

| Режим | Зондирование | Вывод |
|---|---|---|
| `plain`  | раунд 0 всем, ретрай молчунам | `IP alive <ms>` / `IP down` |
| `fping`  | `-C` проб **каждому** хосту | `IP : r0 r1 r2` (`-` = потеря), как `fping -C -q` |
| `zabbix` | `-C` проб каждому хосту, либо `--discover` | `<key>.alive` (0/1), `<key>.loss` (0..1), `<key>.rtt` (с) |
| `json`   | `-C` проб каждому хосту, либо `--discover` | один JSON-массив `[{ip,name,alive,loss,rtt},…]` |

Две стратегии зондирования, независимо от формата вывода:

* **фиксированные `-C` пробы** *каждому* хосту — нужны для измерения loss%. По
  умолчанию в `fping`/`zabbix`/`json`.
* **адаптивный ретрай** — послать батч один раз, затем добить *только молчунов*
  до `-r` раз (исходная идея: быстрый жив/мёртв, без лишнего трафика тем, кто уже
  ответил). Всегда у `plain`; для `zabbix`/`json` включается флагом `--discover`,
  когда нужен только статус (тогда `loss` = 0 или 1).

То есть для чистой проверки живости в Zabbix `fastping -o zabbix --discover -r 2`
пингует каждый хост один раз и повторяет только не ответившим — в отличие от
дефолтного `-C`-режима, где все пингуются N раз.

## Тюнинг sysctl / NIC

На больших батчах узкое место — **потерянные ответы**, которые превращаются в
лишние ретраи. Поднимаем буферы и backlog:

```sh
sysctl -w net.core.rmem_max=16777216
sysctl -w net.core.rmem_default=16777216
sysctl -w net.core.netdev_max_backlog=250000
sysctl -w net.ipv4.icmp_ratelimit=0          # на самом пингующем, опц.
ethtool -G eth0 rx 4096 tx 4096
```

Для реальных свипов всегда задавай `--rate`. Без него можно переполнить свой же
TX-ring и словить ICMP rate-limit на промежуточных роутерах → ложные «down» и
лишние ретраи. Начинай с 20k–50k pps и смотри на потери.

> Сами цели тоже rate-лимитят ICMP (`net.ipv4.icmp_ratelimit`, по умолчанию
> 1000/с). Это и есть главная причина ложных отрицательных результатов за один
> проход — ровно для этого и нужны раунды ретраев.

## Интеграция с Zabbix

Подробно — в [`zabbix/template.md`](zabbix/template.md). Кратко модели:

* **Модель A — trapper + нативный sender.** Один запуск пингует весь батч и пушит
  результаты одной TCP-сессией прямо на сервер/прокси (`--server`). Заводишь
  trapper-айтемы `icmp.alive`, `icmp.loss`, `icmp.rtt`. Финальный пуш бьётся на
  чанки по `--batch` значений на сессию (по умолч. 1000, `0` = одной). Результат
  уходит **один раз после всего свипа** (loss% требует всех `-C` проб на хост).
  Расписание — через [`zabbix/fastping.timer`](zabbix/fastping.timer) или cron.

* **Модель C — одна активная проверка на localhost кормит ВСЕ хосты.** Самый
  «нативный» путь: один айтем (*Zabbix agent (active)*) запускает `fastping -o
  json`, возвращая весь батч одним JSON-массивом; **master item + dependent items
  + LLD** с препроцессингом JSONPath размножают его без повторных запусков и без
  trapper-пуша. UserParameter — в [`zabbix/fastping.conf`](zabbix/fastping.conf).

* **Откуда брать список хостов** — через API
  ([`zabbix/zbx-hosts.sh`](zabbix/zbx-hosts.sh), `host.get`), пишет строки `IP
  host.host` — ровно формат `-f`.

* **Fallback, если данные не пришли** — авто-переключения push→poll в Zabbix нет;
  паттерн: триггер `nodata()` на `icmp.alive` + резервный низкочастотный штатный
  `icmpping` (active, на fping), читаемый когда trapper устарел.

## Интеграция с Nagios / Icinga

Две обёртки в [`nagios/`](nagios/):

**Активная, один хост** — [`nagios/check_fastping`](nagios/check_fastping), почти
drop-in вместо `check_icmp`/`check_ping`:

```
check_fastping -H 8.8.8.8 -w 20,200 -c 100,500 -C 5
# FASTPING OK - 8.8.8.8 rta 2.207ms, lost 0% | rta=2.207ms;200;500;0; pl=0%;20;100;0;100
```

Возвращает OK/WARNING/CRITICAL/UNKNOWN с perfdata `rta`/`pl`. `-w`/`-c` — это
`loss%,rta_ms`. `-b afpacket` — бэкенд AF_PACKET. Один процесс на хост, как
штатные плагины: просто, но без батчинга.

**Батч, пассивно (сохраняет батчинг)** —
[`nagios/fastping-passive.sh`](nagios/fastping-passive.sh) обходит весь список за
один запуск и сдаёт по одному пассивному результату на хост **текстом** во
внешний командный пайп Nagios (FIFO) или демону NSCA по TCP:

```sh
# файл хостов: построчно "ip nagios_host_name"
fastping-passive.sh -f hosts.txt -s ICMP -C 5 -P /usr/local/nagios/var/rw/nagios.cmd
# -> [<ts>] PROCESS_SERVICE_CHECK_RESULT;<host>;ICMP;<rc>;<output|perfdata>
fastping-passive.sh -f hosts.txt -s ICMP --nsca nagios-host:5667
```

Это Nagios-аналог trapper-модели Zabbix: один быстрый батч-свип, много пассивных
результатов. На стороне Nagios сервисы объявляются пассивными. (Нужен `jq`.)

## Roadmap

* `AF_PACKET` + `PACKET_TX_RING` (mmap) для настоящих мульти-Mpps и чтобы не
  терять ответы в буфере сокета.
* Криптографический cookie (SipHash) вместо FNV — защита от спуфинга в открытом
  интернете.
* IPv6 в бэкенде AF_PACKET (нужна ручная ICMPv6-чексумма с pseudo-header);
  параллельный DNS-резолв для больших списков хостнеймов.
* Опциональный `--stream` режим для раннего discovery.
